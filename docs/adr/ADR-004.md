# ADR-004: Better Auth over Clerk (Self-Hosted Authentication)

- **Status:** Accepted
- **Date:** 2026-07-06
- **Blueprint references:** §8.1, §8.5, §3.1.4, §3.1.6
- **Supersedes:** none
- **Superseded by:** none

## Context

Authentication for a multi-tenant clinic SaaS in Algeria must satisfy four
hard requirements:

1. **Data residency** — under Law 18-07 (amended by Law 25-11) and §3.1.6,
   all personal data (including authentication identifiers, session tokens,
   and audit events tied to named users) **must stay in Algeria**. Cross-
   border transfer requires prior ANPDP authorization (a multi-month process).
2. **Multi-tenant organization model** — auth must model "user belongs to
   clinic (organization), with role inside that organization" natively, and
   play nicely with PostgreSQL RLS (ADR-001).
3. **MIT / no vendor lock-in** — the project must remain self-hostable on
   Algerian sovereign infrastructure with zero recurring SaaS spend on auth.
4. **Future patient portal** — Phase 12+ (§11.6) requires a separate auth
   surface (phone + OTP) for patients; the auth library must support that
   without a second auth system.

## Decision

Adopt **Better Auth** (self-hosted) with the **Organization** and **RBAC**
plugins as the sole authentication system.

### Rationale [Blueprint §8.5]

- **Self-hosted**: runs inside the NestJS modular monolith (ADR-002) on
  CERIST/Djezzy Cloud VPS. No data leaves Algeria. Zero vendor lock-in.
- **Organization plugin**: built-in members, teams, roles, and an
  organization switcher — explicitly designed for multi-tenant SaaS. Maps
  cleanly to the `clinic` tenant and the RBAC role hierarchy in §9.2
  (super_admin → clinic_admin → physician/dentist/nurse/receptionist/…).
- **RLS-friendly**: the Better Auth `session` table is a normal Drizzle
  table and can be RLS-protected alongside every other tenant-scoped table.
  Better Auth supersedes the deprecated Lucia (whose maintainer announced
  deprecation in October 2024).
- **MIT-licensed**.
- **Phone-OTP support**: future patient portal (Phase 12+) can use a second
  Better Auth instance configured for phone-number + OTP, sharing the same
  Postgres.

### Integration points

- **Tenant interceptor** (`infrastructure/rls/`): reads the Better Auth
  session's active organization ID and issues `SET LOCAL app.current_tenant`
  per request. RLS does the rest.
- **Audit interceptor** (§9.7): every login, logout, MFA enable, password
  reset, and role change is written to the append-only hash-chained
  `audit_log`.
- **MFA**: TOTP via the Better Auth MFA plugin from day 1 for any role with
  `clinic:manage_users` or `audit:read`.

## Consequences

**Positive**

- Law 18-07 / Law 25-11 data residency satisfied with no cross-border
  transfer authorization needed for auth data.
- One auth library across staff and (future) patient portal.
- Organization switcher is provided, not built.
- Session table participates in RLS → a user in clinic A literally cannot
  read clinic B's session rows at the database level.

**Negative**

- Smaller ecosystem than Clerk or Auth.js; fewer pre-built UI components.
  Mitigated by shadcn/ui-based forms we build ourselves.
- We operate the auth system (rate limiting, MFA backup codes, password
  reset email/SMS plumbing). Mitigated by BullMQ worker for async flows and
  the operational discipline of the modular monolith.

## Alternatives Considered

### Clerk

Managed, US-hosted auth SaaS. Has excellent DX and a mature Organization
model. Rejected because:
- **Data residency blocker**: Clerk is managed and US-hosted. Authentication
  identifiers (email, phone, IP, user agent) are personal data under Law
  18-07. Using Clerk requires prior ANPDP cross-border transfer authorization
  — a multi-month process with no guarantee of approval, and an ongoing
  compliance liability for a clinic SaaS that processes health data.
- **Vendor lock-in**: per-seat pricing scales poorly; vendor outage halts
  logins for every clinic simultaneously.

### NextAuth / Auth.js v5

Self-hostable, MIT-licensed. Pros: huge ecosystem. Cons: no first-class
multi-tenant Organization primitive (we would build it); session model is
JWT- or DB-backed but the DB adapter story is less clean than Better Auth's
Drizzle-native tables. Listed in §8.5 as the **fallback** if Better Auth's
smaller ecosystem becomes a real obstacle.

### Supabase Auth

Hosted Postgres + Auth bundle. Pros: organization/RLS story out of the box.
Cons: ties us to Supabase's hosting model, which conflicts with the
self-hosted-on-Algerian-sovereign-infrastructure posture (§6). Rejected for
the same residency and lock-in reasons as Clerk, plus Supabase's RLS is
designed for their hosted Postgres — we already get RLS from our own
PostgreSQL 17 (ADR-001).
