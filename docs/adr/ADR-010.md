# ADR-010: AI Agent Review Session per PR instead of an Always-On AI PR Review Bot

> **Updated by [ADR-012](./ADR-012.md) (2026-07-07):** the "CI deferred to
> Phase 7" portions of this ADR — specifically the "No machine-enforceable
> gate (yet)" negative consequence and the "until then the author's local
> `pnpm lint && pnpm typecheck && pnpm test` is the gate" sentence in
> "Updated merge gate" — are **rescinded**. CI is now the machine gate
> (`.github/workflows/ci.yml` + `.github/workflows/gitleaks.yml`). The
> manual AI-agent review session this ADR establishes is unchanged and
> remains the second layer of the merge gate. See ADR-012 for the full
> decision and rationale.

- **Status:** Accepted (with the "CI deferred" portions superseded by ADR-012)
- **Date:** 2026-07-06
- **Supersedes:** Roadmap v2.1 §2.6 (Install an AI PR Review Bot)
- **Decider:** Operator (AguHo)

## Context

The Atomic Delivery Roadmap v2.1 §2.6 prescribes installing a third-party AI
PR-review bot — a SaaS that posts automated review comments on every PR open —
to enforce AGENTS.md rules before a human review. Several free-tier options
exist, but each introduces trade-offs that are unfavorable for this project:

- **Data egress.** A third-party bot reads the entire PR diff, which may include
  schema details, tenant-isolation logic, or pseudonymous patient-adjacent code.
  Under Algerian Law 18-07 (amended by Law 25-11), even metadata about the data
  model is sensitive; sending diffs to a non-sovereign SaaS is a (minor) egress
  concern that the EgressGuard pattern (Blueprint §7.3) is designed to prevent.
- **Review depth vs. cost.** Free tiers are shallow; deep reviews cost money,
  breaking the $0 development budget. The roadmap's $0 thesis (see "The $0
  Budget Reality") depends on not paying per-PR review fees.
- **Tool sprawl.** The operator already uses an AI coding agent as the primary
  workforce. Adding a *different* AI tool just for review doubles the toolchain
  surface and the context-switching cost, with no review-quality gain.
- **Trigger model mismatch.** A bot fires automatically on PR open, before the
  operator has decided whether the PR is ready for review. Many agent-generated
  PRs are works-in-progress; auto-reviewing them wastes runs and produces noise.
- **Configurability.** A bot's review prompt is vendor-controlled; the project's
  rules (FDI notation, TVA classes, RLS FORCE, audit hash chain) are
  domain-specific and change as AGENTS.md evolves. A version-controlled,
  in-repo review prompt is more adaptable.

## Decision

**Replace the always-on AI PR review bot with a manually-triggered AI agent
review session run after each PR is opened and before it is merged.**

The operator already uses an AI coding agent as the workforce. That same agent
tool (or a second invocation of it, with a fresh context) is used to review each
PR diff against AGENTS.md. The session is:

1. **Manual** — the operator triggers it; it does not auto-fire on PR open.
2. **Per-PR** — one session per PR, scoped to that PR's diff.
3. **Tool-agnostic** — uses the operator's chosen AI agent tool (same as the
   coding agent), not a third-party SaaS review bot.
4. **AGENTS.md-anchored** — the review prompt is fixed and version-controlled
   in the repo at `docs/runbooks/ai-agent-pr-review.md`. It checks the same
   rules a bot would (RLS, soft deletes, next-intl, logical properties, FDI,
   TVA, no FHIR JSON, no PII in logs, no cross-module internal imports, etc.).
5. **Recorded** — the session's outcome (pass / pass-with-nits / blocked, plus
   the findings table) is posted as a PR comment by the operator, with a link
   to the session transcript if the tool exposes one. The comment is the
   auditable record that a review ran.

### Updated merge gate

A PR merges only when ALL of the following are true:

1. CI is green (`lint`, `typecheck`, `test`, `build` — once Phase 7 lands; until
   then the author's local `pnpm lint && pnpm typecheck && pnpm test` is the
   gate).
2. An **AI agent review session** has run and its outcome is posted as a PR
   comment, with no unresolved `BLOCK`-level findings.
3. One human review (the operator's own review of an agent's PR counts).
4. All conversations resolved.

This replaces the previous gate #3 ("AI-PR-review-bot approval" from Roadmap
§2.6).

## Consequences

**Positive:**

- Zero additional tooling cost — reuses the existing AI agent subscription.
- No third-party egress of PR diffs; stays within the operator's chosen tool,
  which can be run on local or sovereign infrastructure.
- Operator controls *when* to spend a review run: skip WIP PRs, batch-review
  related PRs in one session if the tool allows multi-file context.
- The review prompt is version-controlled alongside AGENTS.md, so it evolves
  in lockstep with the rules. A change to AGENTS.md carries a matching change
  to the runbook checklist in the same PR.
- The review agent has the same context window and tool access as the coding
  agent, enabling deeper review (e.g., it can run `pnpm typecheck` itself, or
  cross-reference the diff against a specific ADR).

**Negative:**

- **Discipline load.** The operator must remember to run the session before
  every merge. A bot is automatic; a manual session is not. Mitigation: the
  `main-protection` ruleset still blocks direct commits to `main`, and the PR
  template includes an "AI Agent Review Session" checklist field that must be
  ticked before merge.
- **No machine-enforceable gate (yet).** Until Phase 7 adds a CI status check
  that parses the PR comment for the session-outcome marker, the gate is
  human-enforced. The operator accepts this for the $0 path; a future
  `ai-review-session` status check can be added in Phase 7 if the manual
  session proves insufficient.
- **Review quality depends on prompt discipline.** If the operator ad-libs the
  review prompt, coverage drifts. Mitigation: the runbook provides a
  copy-paste prompt and a fixed checklist; deviations are recorded in the PR
  comment.
- **No real-time review on push.** A bot reviews on every push; the session
  reviews only when triggered. For a solo operator this is acceptable; for a
  larger team it would be a regression.

## Alternatives Considered

1. **Always-on third-party AI PR review bot (Roadmap §2.6 original).** Rejected
   for the egress, cost, and tool-sprawl reasons above.
2. **GitHub Actions workflow running a local AI review on PR open.** Better than
   a SaaS bot (no third-party egress, runs on GitHub's runners) but requires
   LLM API keys in CI secrets and burns Actions minutes on every PR push.
   Deferred to a future Phase 7 enhancement if the manual session proves
   insufficient.
3. **Human-only review (no AI review at all).** Rejected — the operator is a
   clinician, not a software engineer, and relies on AI to catch AGENTS.md
   violations a human would miss (e.g., a `pr-2` instead of `ps-2`, a hardcoded
   Arabic string, a missing `tenant_id` index, a FHIR resource stored
   internally).
4. **Self-review by the coding agent before push.** Kept as a *first* line of
   defense (the agent runs `pnpm lint && pnpm typecheck && pnpm test` and
   self-checks the PR template checklist), but a *separate* review session with
   fresh context catches what the author agent rationalizes away. Author-blind
   review is the point.
