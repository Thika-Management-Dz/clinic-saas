# ADR-012: CI is the Machine Gate (Rescinds ADR-010's "CI Deferred to Phase 7")

- **Status:** Accepted
- **Date:** 2026-07-07
- **Blueprint references:** Roadmap v2.1 §2.5 (CI/CD), §2.6 (AI PR review bot — superseded by ADR-010), §2.7.3 (solo-operator merge workaround)
- **Supersedes:** the portion of [ADR-010](./ADR-010.md) that says "Until Phase 7 adds a CI status check … the gate is human-enforced" and "CI is green (`lint`, `typecheck`, `test`, `build` — once Phase 7 lands; until then the author's local `pnpm lint && pnpm typecheck && pnpm test` is the gate)".
- **Superseded by:** none
- **Decider:** Operator (AguHo)

## Context

ADR-010 (accepted 2026-07-06) replaced the always-on third-party AI PR review
bot with a manually-triggered AI agent review session per PR. ADR-010
explicitly acknowledged a downside — "No machine-enforceable gate (yet)" —
and deferred the machine gate to "Phase 7" along with the rest of the CI
work. The reasoning was cost discipline: the $0 budget thesis depended on
not burning GitHub Actions minutes until the project had shippable code.

The [23-page critical code review](../audits/2026-07-07-critical-review.pdf)
(2026-07-07, performed against commit `5e588c8`) examined this decision and
reached a sharper verdict. Its sharpest sentence:

> "The manual [review] session is author-blind review; it is not
> author-blind enforcement. The difference matters."

The reviewer's argument, in summary:

1. ADR-010's manual session is necessary but not sufficient. Author-blind
   *review* catches what the author agent rationalizes away. It does not
   catch what *neither* the author nor the reviewer notices — and a single
   human + AI reviewer pair will miss things a deterministic machine gate
   catches every time.
2. The deferral to "Phase 7" was reasonable at the time ADR-010 was
   written, but the critical review identified six P0 blockers (P0-1 through
   P0-6) — and the absence of a machine gate was the common amplifier. A
   machine gate would have automatically caught at least:
   - **P0-1 (no CI)** — by definition.
   - **P0-3 (withTenant SQL injection pattern)** — gitleaks does not catch
     this directly, but a CI lint rule could (and the audit_log + RLS tests
     would surface regressions).
   - **P0-4 (audit hash chain race condition)** — a concurrent-INSERT test
     in CI would have caught this deterministically.
   - **The credential leaks in PR #26 (Task 19-b)** — a gitleaks pre-commit
     hook + CI scan would have blocked the literal Neon password that the
     author agent wrote inline as a grep pattern.
   - **The GitHub Push Protection flag in PR #27 (Task 19-c)** — same.

3. The cost of CI on a Phase 4 codebase that's mostly skeleton is trivial:
   ~2 minutes per run, free for the first 2,000 minutes/month on GitHub
   Free. The deferral was solving a problem the project doesn't have.

The operator concurred with the reviewer's verdict. This ADR rescinds the
"deferred to Phase 7" portion of ADR-010 and lands CI immediately as
PR #28 (Task 20-a).

## Decision

**CI is the machine gate, effective immediately.** A PR merges only when
ALL of the following are true:

1. **CI is green** — specifically, the `lint`, `typecheck`, `test-scripts`,
   `integration`, and `gitleaks` checks from `.github/workflows/ci.yml`
   and `.github/workflows/gitleaks.yml` all pass. These are
   `required_status_check` rules in the main-protection ruleset (ID
   18567129) after this PR lands. (See `docs/runbooks/ci.md` for the
   relax/restore payloads — they must include the required_status_check
   rule going forward.)
2. **An AI agent review session has run** (per ADR-010, unchanged) and its
   outcome is posted as a PR comment with no unresolved `BLOCK`-level
   findings.
3. **One human review** (the operator's own review of an agent's PR counts).
4. **All conversations resolved.**

This is the same merge gate ADR-010 specified, with one change: gate #1 is
now machine-enforced via the `required_status_check` rule, not human-enforced
via the author's local `pnpm lint && pnpm typecheck && pnpm test`.

### What this changes in ADR-010

The following sentences in ADR-010 are **superseded** by this ADR:

- "**No machine-enforceable gate (yet).** Until Phase 7 adds a CI status
  check that parses the PR comment for the session-outcome marker, the gate
  is human-enforced." → No longer accurate. The CI status check is the
  machine gate; the ADR-010 review session is the author-blind review layer.
- "CI is green (`lint`, `typecheck`, `test`, `build` — once Phase 7 lands;
  until then the author's local `pnpm lint && pnpm typecheck && pnpm test`
  is the gate)." → No longer accurate. CI lands in PR #28.

The rest of ADR-010 — the rationale for the manual review session, the
cost/egress arguments against a third-party bot, the review prompt in
`docs/runbooks/ai-agent-pr-review.md` — is unchanged. This ADR adds a
machine gate; it does not remove the human + AI review gate.

### The CI workflow

`.github/workflows/ci.yml` runs four jobs in parallel on every PR open/sync
and on every push to `main`:

| Job            | What it does                                                            | Needs DB? |
|----------------|-------------------------------------------------------------------------|-----------|
| `lint`         | `pnpm lint` (eslint flat config across all 8 workspaces)                | No        |
| `typecheck`    | `pnpm typecheck` (`tsc --noEmit` across all 8 workspaces)               | No        |
| `test-scripts` | `bash tests/test-setup-workstation.sh` (22 bash tests for Phase 1 fix)  | No        |
| `integration`  | Postgres 17 service container + drizzle migrations + FORCE RLS + audit immutability + 19 Vitest tests | Yes |

`.github/workflows/gitleaks.yml` runs the gitleaks action on every PR/push
to scan for hardcoded secrets. The combination of CI + gitleaks + the ADR-010
manual review session is a three-layer gate: machine lint/type/test,
machine secret-scan, human + AI review.

### Solo-operator merge workaround

The main-protection ruleset (ID 18567129) requires 1 approving review +
code-owner review + thread resolution. The operator is solo and cannot
self-approve a PR. The existing workaround (Roadmap §2.7.3) is preserved:
relax the ruleset (set `required_approving_review_count` to 0,
`require_code_owner_review` to false, `required_review_thread_resolution`
to false) → squash-merge → restore the ruleset to full strictness → verify
with a fresh `GET /repos/{owner}/{repo}/rulesets/18567129`. **Going forward
(after PR #28 lands), the relax/restore payloads MUST include the
`required_status_check` rule** so the CI check stays required even during
the brief relax window. The updated payloads are documented in
`docs/runbooks/ci.md`. **Never leave the ruleset in the relaxed state.**

## Consequences

**Positive:**

- Every PR is automatically validated against lint + typecheck + integration
  tests + secret scan, regardless of whether the author remembered to run
  them locally. A regression in RLS, audit-log integrity, or secret hygiene
  is caught by the machine before a human ever reviews.
- The P0-1 blocker (no CI/CD) is closed. The 30-day remediation roadmap's
  30-1, 30-2, 30-6, 30-7, and 30-8 items all land with this ADR.
- Future Phase 5+ work (TenantInterceptor, AuditInterceptor, EgressGuard)
  can be developed against the safety net of CI. A change that breaks RLS
  isolation or audit-log immutability is caught before merge, not after.
- The gitleaks gate prevents the credential-leak pattern that bit PR #26
  (Neon password inline as grep pattern) and PR #27 (Push Protection flag
  on a redacted-token placeholder). Both would have been caught by a
  gitleaks pre-commit hook.
- Renovate (already configured) can now safely automerge patch-level
  dependency bumps if the operator chooses — CI validates the update before
  merge.

**Negative:**

- **Actions minutes consumed.** Each PR consumes ~5-10 minutes of Actions
  time across the 5 jobs. GitHub Free allows 2,000 minutes/month for private
  repos and unlimited for public repos. This project is public, so the
  budget is unlimited — but the operator should monitor if the project
  ever goes private.
- **Slightly slower PR cycle.** The author's local `pnpm lint && pnpm
  typecheck && pnpm test` is still the first line of defense (cheap, ~30s
  on a warm cache). CI adds 2-3 minutes for the parallel jobs. Acceptable
  trade-off for the machine-enforced guarantee.
- **Ruleset relax/restore payloads are more complex.** The payloads must
  include the `required_status_check` rule after PR #28. This is
  documented in `docs/runbooks/ci.md` and the operator (or AI agent) must
  use the new payloads going forward. Getting this wrong leaves the ruleset
  in an inconsistent state.
- **The machine gate catches machine-detectable issues only.** ADR-010's
  manual review session is still required for rule violations that need
  judgment (e.g., a hardcoded Arabic string, a `pr-2` instead of `ps-2`,
  a missing FHIR-export boundary). This ADR does not remove the manual
  session; it adds a machine layer below it.

## Alternatives Considered

1. **Continue deferring CI to Phase 7 (ADR-010's original stance).**
   Rejected by the critical review and the operator. The deferral was
   reasonable at ADR-010's writing but the critical review demonstrated
   concrete harm (PR #26 + PR #27 credential-leak patterns would have been
   caught automatically). The cost of CI on a public repo is zero.
2. **Add only gitleaks, defer the rest to Phase 7.** Rejected — gitleaks
   catches secrets but not RLS / audit-log / type regressions. The
   integration test job is the higher-leverage piece: it runs the
   compliance-gate RLS tests on every PR, which is the whole point of
   having them.
3. **Use a third-party CI service (Travis, CircleCI).** Rejected — GitHub
   Actions is free for public repos, integrates natively with branch
   protection, and requires no additional account. Travis/CircleCI would
   add account overhead with no benefit.
4. **Self-hosted runners on Algerian infrastructure.** Considered for
   Phase 16 (production compliance — diff egress to GitHub's runners is a
   minor concern under Law 18-07). For now, GitHub-hosted runners are
   acceptable because the codebase contains no patient data — only
   schema, code, and test fixtures with synthetic UUIDs. Re-evaluate when
   Phase 5+ adds real user-data fixtures.
5. **Run AI review in CI (ADR-010 alternative #2).** Rejected for the same
   reasons ADR-010 rejected it: LLM API keys in CI secrets, Actions
   minutes burned on every PR push, and the manual session is cheaper +
   deeper. ADR-010's manual review session is preserved unchanged; this
   ADR only adds the deterministic machine gate below it.

## Implementation Notes

- This ADR is implemented in PR #28 (Task 20-a) by:
  - Adding `.github/workflows/ci.yml` (lint + typecheck + test-scripts +
    integration jobs).
  - Adding `.github/workflows/gitleaks.yml` (gitleaks-action).
  - Adding `.pre-commit-config.yaml` (local hygiene).
  - Adding `SECURITY.md` (vulnerability disclosure policy).
  - Pinning the Orthanc docker image to `orthancteam/orthanc:26.6.1` (not
    `:latest`) in `docker-compose.yml`.
  - Adding `docs/runbooks/ci.md` documenting the CI workflow, the relax/
    restore payloads (with the new required_status_check rule), and the
    gitleaks policy.
  - Updating the main-protection ruleset (ID 18567129) to add a
    `required_status_check` rule requiring the `integration` check.
- After PR #28 lands, ADR-010 should be edited to strike the
  "deferred to Phase 7" language and reference this ADR. (Editing
  accepted ADRs is normally avoided per the ADR convention, but the
  ADR-010 file itself can carry an "Updated by ADR-012" note at the top
  without changing the original decision text.) This ADR does that
  editorial update in the same PR.
