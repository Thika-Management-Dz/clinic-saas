# ADR-002: Modular Monolith over Microservices

- **Status:** Accepted
- **Date:** 2026-07-06
- **Blueprint references:** §7.2, §7.3, §7.4, §8.3
- **Supersedes:** none
- **Superseded by:** none

## Context

The Clinic Management SaaS is built by **a single clinician-operator using AI
coding agents on a $0 ongoing budget**. Cross-domain workflows are the norm,
not the exception: a single patient visit touches 5 domains — patient
check-in → appointment started → encounter created → inventory dispensed →
invoice generated. The system must be deployable on one Algerian VPS
(CERIST Cloud or Djezzy Cloud) and operable by one person.

Martin Fowler's "Monolith First" principle is unambiguous:

> "Almost all the successful microservice stories have started with a monolith
> that got too big and was broken up. Almost all the cases where I've heard of
> a system that was built as a microservice system from scratch, it has ended
> up in serious trouble."

Shopify — one of the largest Ruby on Rails codebases in existence — chose to
evolve into a **modular monolith** rather than decompose into microservices.

## Decision

Adopt a **modular monolith**: a single NestJS deployment
(`@nestjs/platform-fastify`) with strict domain-module boundaries.

### Module structure [Blueprint §7.4]

```
src/
  main.ts
  app.module.ts
  modules/
    auth/           # login, sessions, JWT, refresh tokens, MFA
    clinic/         # tenant mgmt, branches, staff, operatory
    patient/        # demographics, search, merge, consent
    appointment/    # calendar, booking, RRULE recurrence, reminders
    encounter/      # visits, vitals, SOAP notes, problems, allergies
    dental/         # odontogram, treatment plans, perio, CDT procedures
    billing/        # invoices, payments, refunds, cash drawer
    inventory/      # pharmacy + dental materials, dispense, alerts
    imaging/        # Orthanc integration, intraoral photos
    audit/          # append-only audit_log writer + readers
    shared/         # primitives, types, Result<T>, Pagination, Money
  infrastructure/
    drizzle/        # drizzle.config.ts, schema/, migrations/
    rls/            # SET LOCAL app.current_tenant interceptor
    postgres/       # connection pool config
    auth/           # Better Auth config + Organization plugin
    egress/         # EgressGuard interceptor (AWS Paris PII blocking)
```

### Module-boundary rules

1. A module exports its public API **only** via `index.ts`. Importing another
   module's internal services is a lint error (enforced by an ESLint
   `no-restricted-imports` rule keyed on `**/src/**`).
2. Each module owns its Drizzle schema file; the root `schema/index.ts`
   aggregates them for Drizzle-Kit migrations.
3. **Cross-module write-side effects go through `EventEmitter2`** (NestJS
   in-process), never via direct cross-module service calls.
4. Event payloads are **transport-agnostic from day 1**: JSON-serializable,
   no class instances, no Symbol keys, no Date objects (use ISO-8601
   strings). A future extraction to Redis Streams or NATS requires **no
   contract changes** — only the transport adapter.

### Why EventEmitter2 over a message bus

For the canonical 5-domain patient visit, an in-process `EventEmitter2.emit`
is ~1000× faster and simpler than a Redis/NATS round-trip, with no
serialization cost. We pay the discipline tax (transport-agnostic payloads)
now so that extraction to a real message bus — if and when a module truly
needs to scale out — is a deployment change, not a rewrite.

## Consequences

**Positive**

- One deployment, one DB connection pool, one CI pipeline.
- One VM/container on CERIST/Djezzy Cloud instead of N microservices × N
  containers.
- Cross-domain workflows (5-domain patient visit) stay in-process — no
  distributed transactions.
- Easier to discover correct module boundaries inside one codebase before
  extracting later.
- The same `EventEmitter2` payload contracts double as documentation of
  cross-domain coupling.

**Negative**

- A memory leak or fatal exception in any module brings down the whole
  process. Mitigated by PM2/systemd restart, health checks, and the worker
  app (`@nestjs/bullmq`) for crash-isolated background work (SMS reminders,
  payment reconciliation, backup verification, audit-log integrity check).
- No language heterogeneity. Acceptable: end-to-end TypeScript is an explicit
  stack goal (§8) for AI-agent productivity.
- Module boundaries must be policed by lint and review — without discipline,
  a monolith rots into a ball of mud. The `index.ts`-only-public-API rule is
  the primary enforcement lever.

## Alternatives Considered

### Microservices from day one

One NestJS service per domain (auth, patient, appointment, encounter,
dental, billing, …), communicating over HTTP or a message bus. Rejected
because:
- **Distributed transactions** across 5 domains for a single patient visit
  are a research-grade problem, not a v1 problem.
- **Operational overhead**: N containers × N health checks × N log streams
  × N connection pools — unworkable for a solo operator.
- **Fowler's monolith-first**: every greenfield microservice story he has
  seen "ended up in serious trouble."
- The modular monolith's `EventEmitter2` payloads are designed for
  extraction; we can pull a module out **when its scale or release cadence
  actually diverges**, with no contract break.

### Serverless functions (one function per route)

Rejected: PWA + NestJS already gives us HTTP scaling; serverless adds cold
starts incompatible with offline-first clinic workflows and complicates
`SET LOCAL app.current_tenant` per-request lifecycle.
