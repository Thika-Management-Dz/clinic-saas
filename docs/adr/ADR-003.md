# ADR-003: Drizzle ORM over Prisma

- **Status:** Accepted
- **Date:** 2026-07-06
- **Blueprint references:** §8.1, §8.4, §7.1.1
- **Supersedes:** none
- **Superseded by:** none

## Context

The data layer must satisfy four hard requirements:

1. **Native PostgreSQL Row-Level Security** support — RLS is the sole
   multi-tenant isolation mechanism (ADR-001). The ORM must support creating
   policies, roles, and `FORCE ROW LEVEL SECURITY` from the schema.
2. **AI-agent productivity** — the operator builds the system with AI coding
   agents; the ORM's schema language must be plain TypeScript so agents can
   refactor with standard TS tooling.
3. **No codegen round-trip** — a codegen step (Prisma's `prisma generate`)
   adds friction to every schema change and is a frequent source of stale
   types in agent-driven workflows.
4. **SQL control** — for clinical workloads (audit-log hash chaining, FTS,
  JSONB perio charting, pgvector later) we need to drop to raw SQL without
  losing type safety.

## Decision

Adopt **Drizzle ORM** with native RLS support as the sole ORM for PostgreSQL
17. Pair with `drizzle-kit` for migrations (applied in a pre-deploy CI job,
**never** on app boot — see AGENTS.md Do-NOT list).

### Rationale

- **Plain-TS schema**: `pgTable()` definitions are TypeScript. Types are
  *inferred*, not *generated*. `drizzle-kit generate` produces SQL
  migrations only; no client codegen step. An AI agent (or a human) edits
  the schema with standard TS tooling — no DSL, no `schema.prisma` file, no
  `prisma generate` to forget.
- **Native RLS**: Drizzle's `pgPolicy()` API creates RLS policies from the
  schema, so `FORCE ROW LEVEL SECURITY` is checked into version control
  alongside the column definitions — not bolted on by a hand-written
  migration that can drift.
- **SQL control**: `db.execute(sql\`…\`)` and the query builder compose
  cleanly; for the audit-log hash chain (§9.7) we use a raw SQL trigger
  function written in a Drizzle migration, fully typed at the call site.
- **License**: Apache-2.0, permissive, no vendor lock-in.

### Schema ownership

Each NestJS domain module (ADR-002) owns its own Drizzle schema file
(`modules/<domain>/schema/*.ts`). The root `infrastructure/drizzle/schema/index.ts`
re-exports them all for `drizzle-kit` to consume.

## Consequences

**Positive**

- No codegen step → faster inner loop for agents and humans.
- RLS policies live next to the columns they protect → fewer drift bugs.
- Smaller bundle, fewer runtime dependencies than Prisma.
- Migration files are plain SQL → reviewable in PRs, debuggable with `psql`.

**Negative**

- Smaller ecosystem of third-party plugins than Prisma (no equivalent of
  `prisma-erd-generator`, fewer community middlewares). Acceptable at MVP.
- No built-in transparent nested-write API like Prisma's
  `include`/`select` for deep joins — must compose with `with()` and
  `leftJoin()`. AI agents handle this well with TS tooling.
- Drizzle Studio is less polished than Prisma Studio for ad-hoc data
  inspection. Mitigated by `psql` + `pgcli`.

## Alternatives Considered

### Prisma

The dominant TS ORM. Rejected because:
- **RLS workarounds**: Prisma does not natively support Postgres RLS policies
  from the schema; policies must be added by hand-written migrations that
  drift from `schema.prisma`. The Prisma client also runs queries as the
  connection user, requiring `SET LOCAL` to be issued via `$executeRaw`
  before every transaction — fragile.
- **Codegen step**: `prisma generate` must run after every schema change;
  stale generated clients are a recurring AI-agent failure mode.
- **Heavier**: the query engine is a Rust binary shipped via
  `@prisma/client`; larger deploy image, more moving parts.
- **SQL control**: dropping to raw SQL via `$queryRaw` loses type inference.

### Kysely

A pure-TS query builder (no schema, no ORM). Pros: maximal SQL control, no
codegen. Cons: no schema-as-source-of-truth — migrations must be hand-written
and there is no compile-time guarantee that a query references existing
columns. For a multi-tenant RLS-heavy schema, Drizzle's `pgTable()` + `pgPolicy()`
gives us the type-safety Kysely lacks. Kysely is the right choice for
read-heavy analytics; it remains an option for the future analytics warehouse
extract (§12.4).
