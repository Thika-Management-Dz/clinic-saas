# ADR-005: Dexie + Manual Sync for v1, PowerSync for v2

- **Status:** Accepted
- **Date:** 2026-07-06
- **Blueprint references:** §10.1, §10.1.1, §10.1.2, §10.3, §10.4
- **Supersedes:** none
- **Superseded by:** none

## Context

Clinics in Algeria operate with intermittent connectivity: ADSL/4G failovers,
regional outages, and per-clinic DSL lines that drop several times a week.
The doctor at the chair-side tablet and the receptionist at the front desk
must both keep working through outages. The offline-first architecture must
be **safe** (no lost writes), **simple** (operable by one developer), and
**self-hostable on Algerian infrastructure**.

The conflict-resolution surface is small in practice: ~95% of clinical
records (appointments, invoices, odontogram, vitals) have a **single author**
per record. Multi-author records (two doctors co-writing a SOAP note) are
the exception, not the rule.

## Decision

Two-phase offline-first architecture.

### v1 — Dexie (IndexedDB) + manual sync outbox

- **Client store**: Dexie (IndexedDB) is the source of truth on the client.
  TanStack Query reads from Dexie; UI writes go to a Dexie `outbox` table.
- **Sync outbox** (§10.3): every write is queued as an outbox entry
  `{entity_type, entity_id, operation, payload, client_id, created_at,
  attempts, last_error, status}`. The outbox drains on the `online` event
  AND via the Background Sync API in Chromium (progressive enhancement only
  — see §10.2.1; Safari support is partial).
- **Conflict resolution — Last-Write-Wins (LWW)** with server-authoritative
  `updated_at` and tenant-scoped `client_id` (§10.1.2). On a 409 CONFLICT,
  the client re-pulls the server's version and discards its local write.
  Used by Dynamo and Cassandra; acceptable at clinic scale where each record
  typically has a single author.
- **Multi-author fields**: SOAP notes and shared treatment plans can
  optionally wrap their free-text body in a Yjs document for true CRDT
  merging (§10.1.2). Vector clocks are explicitly rejected as too complex
  for the benefit at this scale.
- **Schema designed PowerSync-compatible from day 1**: every table has
  `id UUID PK`, `tenant_id`, `updated_at`, `deleted_at`. No rework needed
  for v2.

### v2 — PowerSync Open Edition

When any of the following triggers is met (§10.4), migrate to PowerSync:

| Trigger condition | Threshold |
|---|---|
| Concurrent editors per patient record | **> 5 per clinic** |
| Manual conflict-resolution effort | **> 1 hour / week** |
| Data-divergence incidents | **observed** (two devices showing different patient states after sync) |

PowerSync is **Postgres-native**, **FSL-1.1-ALv2-licensed** (auto-converts
to Apache-2.0 after 2 years), and **self-hostable on Algerian sovereign
infrastructure**. The migration is a 2–3 week project; see
`docs/runbooks/dexie-to-powersync-migration.md` for the full runbook.

## Consequences

**Positive (v1)**

- Lowest operational complexity for a solo developer — no extra infra to run
  (no sync server, no MongoDB, no separate Postgres replica for sync).
- Sufficient for 2 clinics × ~120 patients/day.
- IndexedDB is the right persistence layer for a Next.js PWA; no mobile-only
  assumptions.
- Schema is already PowerSync-shaped → v2 migration is a client-side swap,
  not a schema rewrite.

**Positive (v2)**

- True CRDT-grade sync for the rare multi-author cases that LWW handles
  poorly.
- PowerSync handles the outbox natively → we delete the foreground drain
  logic.

**Negative**

- **LWW risk**: silent discard of concurrent writes and clock-skew issues.
  Acceptable at clinic scale with single-author records; the multi-author
  fields opt into Yjs CRDTs.
- **Manual sync is real code we maintain** (drain loop, retry/backoff,
  dead-letter table). Paid down by PowerSync in v2.
- **v2 migration is non-trivial** (2–3 weeks, dual-write period, cutover,
  rollback plan — see the runbook).

## Alternatives Considered

### Replicache

Client-only sync library; you must build the push/pull backend yourself.
Highest operational complexity for a solo dev. Rejected for v1; the v1
manual outbox is simpler because the conflict surface is small.

### ElectricSQL

Pivoted in July 2024 to **read-only sync** (writes go through your API).
Insufficient for offline clinic workflows where the doctor must write to the
database while offline. Rejected outright.

### WatermelonDB

Mobile-first (React Native), uses LokiJS (in-memory) on web. Poor fit for a
Next.js PWA where IndexedDB is the right persistence layer. Rejected.

### Yjs / Automerge CRDTs (as the general sync mechanism)

Overkill for structured clinic data (appointments, invoices, odontogram,
vitals) where each record typically has a single author. CRDTs are
appropriate **only for collaborative free-text** (e.g., two doctors co-writing
a SOAP note in real time) — used at the field level, not the row level. See
§10.1.2.

### PowerSync from day one

Rejected as premature for v1: would require running a PowerSync Service +
MongoDB metadata VPS from week 1, plus designing sync rules before we know
the access patterns. The trigger thresholds above are the deliberate
"evaluate now" signal.
