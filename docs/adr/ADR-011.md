# ADR-011: Secrets Management — Gitignored `.env` Files for Local/Staging, GitHub Actions Encrypted Secrets for CI

- **Status:** Accepted
- **Date:** 2026-07-07
- **Blueprint references:** §3 (Data Residency), Roadmap v2.1 §1.5.2 (Doppler CLI), §4.7 (Neon Staging), §2.6 / ADR-010 (Phase 7 CI)
- **Supersedes:** the implicit assumption in Roadmap v2.1 §1.5.2 / §4.7.2 that Doppler would be the canonical secret store from Phase 4 onward
- **Superseded by:** none
- **Decider:** Operator (AguHo)

## Context

Roadmap v2.1 §1.5.2 specifies Doppler as the project's secrets manager ("secrets management dev/staging/prod"). The Phase 4 PR F (`docs/runbooks/neon-staging.md`, merged in PR #25) instructed the operator to store the Neon staging credentials in Doppler using `doppler secrets set` driven by a Doppler **service token**.

Task 19-b (this session) attempted to execute that plan and discovered that **Doppler Service Tokens are gated behind the Doppler Team plan (~$99/month)**. The operator's Doppler workspace (`Thika-Management-Dz`) is on the free Developer plan. The screenshot at `pasted_image_1783405031698.png` (attached to the chat that produced this ADR) shows the workplace Tokens page with "Service Accounts" labeled "Team Feature" and "Service Tokens" empty with the same upgrade gate.

This means the headless, AI-agent-friendly workflow the previous session assumed (`doppler run --token dp.stg.xxx -- pnpm test`) is not available on the free tier. The free tier supports `doppler login` (browser OAuth) for interactive local use, but that flow is impossible in a headless AI-agent sandbox and does not authenticate future AI sessions.

The project needs a secrets-management posture that:

1. Works for **local dev** (`docker compose up` + `pnpm dev` on the operator's machine).
2. Works for **staging** (Neon free tier in Frankfurt, no real patient data).
3. Works for **AI-agent sandboxes** (this session and future ones) — no browser OAuth.
4. Works for **CI** when it lands in Phase 7 (GitHub Actions on `ubuntu-latest` runners).
5. Does not lock in a paid third-party dependency before the project has revenue.
6. Keeps the door open for Doppler (or alternatives) if/when the operator chooses to upgrade.

## Decision

We adopt a **tiered secrets posture**, deliberately splitting local/staging from CI:

### Tier 1 — Local dev + staging (now): gitignored `.env` files

- The repo commits **templates only** (`.env.example`, `.env.staging.example`). Real values live in gitignored files (`.env.local`, `.env.staging`) populated by the operator or by an AI-agent sandbox from credentials shared in chat.
- `.gitignore` already excludes `.env`, `.env.local`, `.env.*.local`, `.env.staging`, `.env.production`. No change to the gitignore pattern is required.
- Files are `chmod 600` (owner read/write only).
- For AI-agent sessions: the agent writes `/tmp/.env.neon` (session-only) during bootstrap, then optionally copies it to `<repo>/.env.staging` (gitignored, persistent across `pnpm` invocations) for the duration of the session. The agent must `shred` / `rm` these files at session end if the operator requests it.
- The `DATABASE_URL` / `MIGRATION_DATABASE_URL` / `DIRECT_URL_STAGING` pattern from Phase 4 is preserved unchanged — only the source of the values changes (Doppler → `.env` file).

### Tier 2 — CI (Phase 7): GitHub Actions encrypted secrets

- When CI lands in Phase 7, GitHub Actions **encrypted repository secrets** will be the canonical source for CI-runtime secrets (`DATABASE_URL`, `MIGRATION_DATABASE_URL`, `BETTER_AUTH_SECRET`, etc.).
- GitHub Actions secrets are free, scoped to the repo, masked in logs, and accessible to workflows via `secrets.*` context — no third-party dependency.
- This pairs naturally with the existing GitHub PAT (`GH_TOKEN`) the operator already uses for AI-agent sessions; the same operator-managed surface area.
- CI workflows will load secrets into the runner's environment via the `env:` block, then run `pnpm test` / `pnpm typecheck` / `pnpm lint` exactly as the operator does locally. No `doppler run` wrapper required.

### Tier 3 — Production (Phase 16): TBD, likely on-host `.env` + systemd `LoadCredential`

- Production runs on Algerian sovereign infrastructure (CERIST Cloud or Djezzy Cloud VPS, per Blueprint §3). There is no third-party SaaS secrets manager in the production loop — that would itself be a data-residency concern.
- The production secrets posture will be decided in Phase 16. Likely shape: on-host `/etc/clinic-saas/env` (chmod 600, owner `clinic-saas`, mode locked down), loaded into the NestJS service via systemd `LoadCredential=` (passes secrets as files in `/run/credentials/clinic-saas.service/`, never written to disk). Backups go to a physical safe (per Roadmap §15.5.5) — not to any cloud secrets store.

### Doppler: deferred, not rejected

- The operator may still choose to upgrade to Doppler Team in a future phase if the headless service-token workflow becomes valuable (e.g., for staging deploys from a non-GitHub-Actions context, or for sharing secrets across multiple deployment targets).
- If Doppler is adopted later, this ADR will be superseded by a new one (ADR-012 or later) that specifies the migration path. The `.env`-file pattern is forward-compatible: switching to `doppler run` is a one-line wrapper change in any script that currently sources `.env`.
- This ADR does **not** remove Doppler from `.env.example` or `docs/runbooks/workstation-setup.md` — the CLI is still installed per the roadmap, and `doppler login` (interactive) still works on the operator's local machine. We are only declining to use it as the headless/AI/CI secret store for now.

## Alternatives Considered

| Alternative | Why not (for now) |
|---|---|
| **Doppler Team (~$99/mo)** | Paid. The project is pre-revenue. The headless service-token workflow is the only thing we'd be paying for, and GitHub Actions secrets covers the same need for free in Phase 7. Re-evaluate post-revenue. |
| **`doppler login` (free, browser OAuth)** | Works on the operator's local machine, but impossible in headless AI-agent sandboxes and not portable across sessions. Doesn't solve the AI/CI use case. |
| **1Password CLI** | Excellent for the operator personally, but adds a paid dependency (`op` CLI + 1Password subscription) for the same headless-secret-injection use case GitHub Actions covers for free. Could be revisited for production in Phase 16. |
| **HashiCorp Vault** | Massive overkill for a 2-clinic SaaS. Operations burden exceeds the value. |
| **Infisical (open-source Doppler alternative)** | Self-hostable, free, and supports service tokens — but adds an operations surface (a service to keep running, back up, secure) that the operator (solo) doesn't need yet. Re-evaluate if/when team grows beyond 1. |
| **AWS Secrets Manager / GCP Secret Manager** | Wrong cloud (Algerian data residency requires sovereign infra). Also paid per secret per month. |

## Consequences

### Positive

- **Zero new paid dependencies.** The project stays on the free tier (Neon Free, GitHub Free, Doppler Free) until revenue justifies upgrades.
- **AI-agent-friendly.** Future AI sessions can write `.env.staging` from credentials shared in chat — no browser OAuth, no Doppler token to manage.
- **CI-ready.** Phase 7 CI uses GitHub Actions secrets natively — no `doppler run` wrapper, no extra auth step in the workflow.
- **Forward-compatible.** Switching to Doppler later (if the operator upgrades) is a one-line wrapper change. Switching to on-host `.env` + systemd in production is also a one-line change.
- **Compliance-friendly.** Production never touches a third-party SaaS secrets store — secrets stay on Algerian sovereign infrastructure.

### Negative

- **Secret rotation is manual.** No `doppler secrets rotate` convenience. The operator must `ALTER ROLE … PASSWORD …` on Postgres, then update `.env.staging` + GitHub Actions secrets in two places. This is acceptable for a solo project; would become painful at team size >3.
- **No audit log of secret access.** Doppler logs every secret read; `.env` files do not. Mitigation: the `audit_log` table itself is the compliance-relevant audit trail (every mutation is hash-chained); secret-access audit is a Tier-2 concern that GitHub Actions logs partially cover for CI.
- **Two sources of truth for staging secrets.** The operator's local `.env.staging` and the GitHub Actions secrets can drift. Mitigation: Phase 7 CI workflow will document the "rotate both" procedure in `docs/runbooks/ci-secrets.md` (to be written when CI lands).
- **`.env` files are easier to leak by accident than a secrets manager.** Mitigation: `.gitignore` already excludes them; a `pre-commit` hook to scan for secrets (Phase 7) will add a second layer; the operator's `git diff` discipline is the first layer.
