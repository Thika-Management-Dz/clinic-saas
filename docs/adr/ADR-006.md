# ADR-006: Lean Schema + Future National-Interop Adapter (FHIR as First Implementation)

- **Status:** Accepted
- **Date:** 2026-07-06
- **Blueprint references:** §9.1, §9.1.1, §9.3, §11.2
- **Supersedes:** none
- **Superseded by:** none

## Context

Algeria's national health interoperability backbone (DEM.DZ) does not yet
ship a public API. When it does, it may be FHIR R4/R5, FHIR R6, HL7 v2, or
something Algeria-specific. We must not bet the internal storage model on
any one of these.

The clinic SaaS's internal information model is based on the **OpenMRS
Patient → Encounter → Observation** spine (§9.3), which is simpler than FHIR,
battle-tested in low-resource settings, and maps cleanly to FHIR resources
when needed.

## Decision

**Do NOT store FHIR JSON internally.** Design lean Drizzle tables that map
1:1 to FHIR resources, and add a `NationalInteropAdapter` interface with
`FhirAdapter` as the first implementation and `FutureFormatAdapter` as a
placeholder for whatever DEM.DZ eventually ships.

### Rationale [Blueprint §9.1.1]

FHIR-as-canonical-model is rejected for the storage layer because:

1. **Verbosity**: a FHIR `Patient` resource has 50+ top-level fields, most
   unused for a dental clinic — ~3–5× storage overhead vs a lean Drizzle
   table.
2. **Release cadence**: FHIR has a 2-year major-release cycle
   (R3 → R4 → R4B → R5 → R6 ballot); each release deprecates fields and
   adds new ones. If FHIR is your storage model, you must migrate with each
   release.
3. **No multi-tenancy primitive**: FHIR has no native `tenant_id`; you must
   bolt on extensions or partition via FHIR compartments — fighting the
   standard, not using it.
4. **Round-trip cost**: creating a single Encounter resource often requires
   first creating/finding Practitioner, PractitionerRole, Organization,
   Location, Patient, Coverage resources — **7 round-trips for one
   appointment**.
5. **Learning curve**: 2–4 weeks for a TypeScript engineer.

Instead, lean Drizzle tables map 1:1 to FHIR resources:

| Drizzle table | FHIR resource |
|---|---|
| `patient` | `Patient` |
| `encounter` | `Encounter` |
| `observation` (vital, lab, perio, imaging-finding) | `Observation` |
| `problem_list` | `Condition` |
| `allergy` | `AllergyIntolerance` |
| `medication` | `MedicationRequest` |
| `immunization` | `Immunization` |
| `lab_result` | `Observation` + `DiagnosticReport` |
| `appointment` | `Appointment` |
| `invoice` | (no direct FHIR equivalent; `Claim`/`Invoice` partial) |
| `audit_log` | `AuditEvent` |

### Adapter pattern

```ts
// packages/contracts/src/national-interop/types.ts
export interface NationalInteropAdapter {
  exportPatient(patientId: UUID): Promise<FhirPatient | FutureFormatPatient>;
  exportEncounter(encounterId: UUID): Promise<FhirEncounter | FutureFormatEncounter>;
  // ... one method per resource type we expose to national systems
}

export class FhirAdapter implements NationalInteropAdapter { /* … */ }
export class FutureFormatAdapter implements NationalInteropAdapter { /* … */ }
```

The adapter lives at the **integration boundary** — invoked only when
DEM.DZ (or a lab, or another clinic system) requests data. The internal
storage layer is unaware of FHIR. This gives us interoperability with DEM.DZ
when it ships an API **without paying the verbosity tax internally**.

Firely's 2025 "State of FHIR" report shows 71% of respondents use FHIR
actively in their country — the trend is clear, but the adoption for a
startup should be **at the integration boundary, not the storage layer**.

## Consequences

**Positive**

- Storage stays lean and fast; no 3–5× overhead, no 2-year migration tax.
- RLS-based multi-tenancy (ADR-001) applies uniformly — no FHIR compartment
  gymnastics.
- 7-round-trip-per-encounter cost paid only when an external system actually
  requests the export, not on every write.
- Hedged against DEM.DZ shipping a non-FHIR API: `FutureFormatAdapter` is a
  new class, not a storage migration.
- Internal schema can evolve with clinic needs; the adapter absorbs FHIR
  version drift.

**Negative**

- Adapter is real code we maintain — every resource type needs an
  export method.
- Outbound FHIR conformance must be tested (e.g., against the Firely .NET
  validator or the official HL7 FHIR Validator Java CLI) — added to CI.
- Mapping edge cases (e.g., FHIR `Observation.component` for vitals) require
  thought; documented per-resource in the adapter's JSDoc.

## Alternatives Considered

### FHIR-as-canonical-model

Store FHIR resources as JSONB in Postgres (HAPI-on-Postgres style). Rejected
for the five reasons above (verbosity, release cadence, no tenant primitive,
round-trip cost, learning curve). Re-evaluate only if DEM.DZ mandates FHIR
storage as a compliance condition — currently no such mandate exists.

### OpenMRS-as-canonical-model (run the OpenMRS Java platform)

Rejected: OpenMRS is a Java platform; our stack is end-to-end TypeScript
(§8). We adopt the **OpenMRS information model** (Patient → Encounter →
Observation), not the OpenMRS platform.

### HAPI FHIR server as the API surface

Optional future addition: run a HAPI FHIR server alongside the NestJS
modular monolith, fed by the `FhirAdapter`. Deferred until DEM.DZ actually
requires a FHIR endpoint; the adapter interface is designed to plug into
either an in-process NestJS controller or a separate HAPI server.
