# ADR-001: Pool-Model Multi-Tenancy with PostgreSQL Row-Level Security

- **Status:** Accepted
- **Date:** 2026-07-06
- **Blueprint references:** §7.1, §7.1.1, §9.1
- **Supersedes:** none
- **Superseded by:** none

## Context

The Clinic Management SaaS targets **2 → 50 clinics** at MVP scale, with
**~120 patients/day/clinic** and dozens of users per tenant (reception,
physicians, dentists, nurses, billing). All clinical data is **sensitive
personal data** under Algerian Law 18-07 (amended by Law 25-11) and must be
tenant-isolated with defense-in-depth: even a forgotten `WHERE tenant_id = …`
clause in the application layer must not leak data across clinics.

AWS SaaS Factory defines three canonical multi-tenancy models:

1. **Silo** — one database per tenant.
2. **Bridge** — one shared database, separate schema per tenant.
3. **Pool** — one shared database, one shared schema, with a `tenant_id`
   discriminator on every row and Row-Level Security (RLS) enforcing
   isolation at the database engine.

Crunchy Data's Postgres-specific framing concurs: "tenant discriminator in
shared tables" with `tenant_id` on every table is the right default;
DB-per-tenant becomes unwieldy at 10+ tenants due to migration overhead, and
schema-per-tenant offers minimal isolation gain over RLS for substantial
operational cost.

The application runs on **Algerian sovereign infrastructure** (CERIST Cloud
or Djezzy Cloud VPS — see §6), where per-tenant VMs and operational overhead
are disproportionately expensive relative to clinic revenue. A solo operator
using AI coding agents cannot maintain 50 separate databases.

## Decision

Adopt the **Pool Model** with **PostgreSQL Row-Level Security** as the sole
multi-tenancy mechanism.

### Schema pattern (every tenant-scoped table)

```sql
CREATE TABLE patient (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id   UUID NOT NULL REFERENCES clinic(id),
  -- ... domain columns ...
  created_at  TIMESTAMPTZ NOT NULL DEFAULT clock_timestamp(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT clock_timestamp(),
  deleted_at  TIMESTAMPTZ,
  created_by  UUID REFERENCES app_user(id)
);
ALTER TABLE patient ENABLE ROW LEVEL SECURITY;
ALTER TABLE patient FORCE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON patient
  USING      (tenant_id = NULLIF(current_setting('app.current_tenant', true), '')::uuid)
  WITH CHECK (tenant_id = NULLIF(current_setting('app.current_tenant', true), '')::uuid);
CREATE INDEX patient_tenant_idx      ON patient (tenant_id) WHERE deleted_at IS NULL;
CREATE INDEX patient_tenant_name_idx ON patient (tenant_id, family_name, given_name);
```

### Mandatory configuration points [Blueprint §7.1.1]

1. `ENABLE ROW LEVEL SECURITY` on every tenant-scoped table.
2. `FORCE ROW LEVEL SECURITY` on every tenant-scoped table — without `FORCE`,
   the table owner bypasses policies, and Drizzle migrations run as table
   owner.
3. `CREATE INDEX … USING btree(tenant_id)` on every tenant-scoped table.
   (Supabase benchmarks show 171 ms → <0.1 ms with the index; the operator
   MUST re-benchmark on CERIST Cloud VPS to confirm.)
4. Reserve exactly one named `ops_superuser` role with `BYPASSRLS`; the
   application DB role MUST NOT have `BYPASSRLS`.
5. Wrap policy helper functions in `(select …)` so Postgres treats them as
   stable scalars (Supabase measured 173 000 ms → 16 ms).
6. RLS does **not** apply to `TRUNCATE` or `REFERENCES` — `REVOKE TRUNCATE`
   from the application role.
7. PgBouncer in `transaction` pooling mode breaks `SET LOCAL` session vars —
   use `pool_mode=session`, or adopt Supavisor/PgCat which support session
   vars in transaction mode.

### Tenant context propagation

The NestJS request lifecycle issues `SET LOCAL app.current_tenant = $1` at the
start of every transaction via a tenant interceptor (`infrastructure/rls/`).
RLS policies read `current_setting('app.current_tenant')` and filter every
query. The session table of Better Auth is also RLS-protected (see ADR-004).

### Citus sharding trigger threshold

Citus sharding is **premature** at this scale but documented as a deliberate
escape hatch. **Trigger evaluation of Citus when ANY of the following is
true:**

| Trigger condition | Threshold |
|---|---|
| Tenant count | **> 200 tenants** |
| Database size | **> 5 TB** |
| p95 API latency | **> 500 ms** sustained over a 1-week window |

Crossing any threshold does NOT mandate migration; it mandates *evaluation*,
recorded in a new ADR.

## Consequences

**Positive**

- One database, one schema, one connection pool, one CI pipeline —
  operationally tractable for a solo operator.
- Tenant isolation enforced at the database engine, not in application code.
  A bug in a `WHERE` clause cannot leak data.
- Migrations apply once across all tenants — no per-tenant migration fan-out.
- New tenants are a single `INSERT INTO clinic` row; no provisioning step.
- Schema designed PowerSync-compatible from day 1 (`id` UUID PK,
  `tenant_id`, `updated_at`, `deleted_at` on every table — see ADR-005).
- Backups (§13) capture all tenants in one pgBackRest run.

**Negative**

- One noisy tenant can degrade performance for all tenants; mitigated by
  per-tenant rate limiting at the NestJS layer and connection-pool caps.
- Cross-tenant analytics require `SET LOCAL row_security = off` from the
  `ops_superuser` role, with audit-log entries (§9.7).
- Operator must keep discipline on `FORCE ROW LEVEL SECURITY` for every new
  table — verified by a CI test that asserts no tenant-scoped table lacks
  FORCE+policy (see `docs/conventions/testing.md`).
- `SET LOCAL` is per-transaction; any code path that runs SQL outside the
  NestJS request lifecycle (e.g., a BullMQ worker) must also set
  `app.current_tenant` from the job payload.

## Alternatives Considered

### Silo — database-per-tenant

One PostgreSQL database per clinic. Strong isolation (different databases,
different roles, different backup schedules). Rejected because:
- Migration fan-out: 50 tenants × N migrations = 50× operational cost.
- Connection pool exhaustion: each NestJS process must hold connections to
  every tenant DB; with 50 tenants × 10 connections = 500 idle connections.
- Backups, monitoring, and PITR must be configured per-tenant.
- Becomes unwieldy at 10+ tenants per AWS SaaS Factory and Crunchy Data.

### Bridge — shared database, schema-per-tenant

One database, one schema per clinic (`clinic_42.patient`). Some isolation
gain over RLS. Rejected because:
- Minimal isolation gain over RLS for substantial operational cost (must
  `SET search_path` per request, must run migrations against every schema).
- Still one connection pool but with per-schema search_path bookkeeping.
- PowerSync sync rules are simpler with a single schema + `tenant_id`.

### Citus sharding from day one

Distribute tenant rows across shards by `tenant_id` hash. Rejected as
premature optimization at 2 → 50 tenants; trigger threshold documented above.
