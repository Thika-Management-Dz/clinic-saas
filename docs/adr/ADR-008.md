# ADR-008: Chargily Pay for MVP Payments

- **Status:** Accepted
- **Date:** 2026-07-06
- **Blueprint references:** §4.1, §4.1.1, §4.1.2, §4.1.3, §9.6
- **Supersedes:** none
- **Superseded by:** none

## Context

The Algerian payment landscape centers on **SATIM** (Société Algérienne des
Transmissions Interbancaires), the state-owned interbank payment switch. Two
card schemes dominate:

- **CIB** (Carte Interbancaire) — issued by Algerian commercial banks.
- **Edahabia** — issued by Algérie Poste; the majority of active cards in
  Algeria.

Two integration paths exist:

1. **SATIM direct** — requires opening a merchant account with a
   SATIM-supporting Algerian bank (BNA, CPA, BEA, BADR, …), submitting
   onboarding documents via the SATIM portal at `cibweb.dz`, and negotiating
   an enterprise contract. **Onboarding takes months.** SATIM's public API
   has **no recurring or card-on-file tokenization** — plan for SATIM-direct
   recurring as not available.
2. **Chargily Pay v2** — Algerian payment gateway aggregating CIB + Edahabia
   behind a single REST API. Onboarding takes **days**.

The clinic also collects **cash** at reception and runs a **physical TPE**
(in-clinic card terminal) — these are non-negotiable for an Algerian clinic,
regardless of online payment integration.

## Decision

Adopt **Chargily Pay v2** as the MVP online payment provider, behind a
**`PaymentProvider` strategy pattern** so additional adapters are
configuration changes, not code changes.

### Rationale [Blueprint §4.1.2]

| Tier | Monthly fee | Commission | Caps | Use case |
|---|---|---|---|---|
| Startup | Free | **0%** | None | First month for any new account; indefinite for very low volume |
| Comfort | $49/mo | 1.25% | None | MVP default after first month if volume > ~$4,000/mo |
| Supreme | $99/mo | 2.5% | None | High-volume clinics with premium support needs |

- **Algerian** — supports the two Algerian card schemes (CIB + Edahabia).
- **Startup tier = 0% commission** for the first month on any new account.
- **Onboarding in days**, not months.
- **No per-transaction caps** on any tier.

### Payment layer — strategy pattern [Blueprint §4.1.3]

```ts
// packages/contracts/src/billing/payment-provider.ts
export interface PaymentProvider {
  initiatePayment(input: InitiatePaymentInput): Promise<InitiatePaymentResult>;
  verifyPayment(reference: string): Promise<PaymentStatus>;
  refund(reference: string, amount: Money): Promise<RefundResult>;
}

export class ChargilyPayAdapter implements PaymentProvider { /* … */ }
export class SlickPayAdapter     implements PaymentProvider { /* … */ }  // backup
export class MizaniyaPayAdapter  implements PaymentProvider { /* … */ }  // backup
export class CashAdapter         implements PaymentProvider { /* … */ }  // always available at reception
export class TPEAdapter          implements PaymentProvider { /* … */ }  // in-clinic terminal
```

The `billing` module dispatches to the configured provider per `payment.method`
(`cash | cib | edahabia | cheque | bank_transfer | other` — see §9.6 schema).
If Chargily Pay has an outage, the operator can fall back to SlickPay or
Mizaniya Pay, and cash + physical TPE are **always** available at reception.

### Resilience posture

- Cash and physical TPE are **mandatory** at reception regardless of online
  integration. The Algerian patient population uses cash predominantly; the
  online integration serves patients who pay invoices remotely (e.g.,
  patient-portal invoice payment, Phase 12+).
- Chargily Pay is the **default**; SlickPay and Mizaniya Pay are **documented
  backups** (custom pricing, days-onboarding) selectable by config.

## Consequences

**Positive**

- MVP ships with online payments in days, not months.
- 0% commission during the first month lets us validate the workflow with
  real money before paying any commission.
- Strategy pattern → swapping providers is a config change.
- Cash + TPE always available → no single point of failure for reception.

**Negative**

- Chargily Pay is a third-party aggregator; we depend on their uptime and
  their SATIM integration. Mitigated by SlickPay / Mizaniya Pay / cash / TPE
  fallbacks.
- No native recurring / card-on-file tokenization (same limitation as SATIM
  direct). Subscription billing, if ever needed, must be implemented as
  scheduled `initiatePayment` calls — not true card-on-file. Acceptable: the
  clinic SaaS is **out-of-pocket B2C** per §9.6, not subscription B2C.
- Chargily Pay processing fees on Comfort/Supreme tiers (1.25% / 2.5%) —
  re-evaluate against SATIM direct once monthly volume justifies the
  enterprise contract overhead.

## Alternatives Considered

### SATIM direct

~1.5–2% CIB, ~1–1.5% Edahabia. Branded SATIM payment page in FR/EN/AR.
Rejected for MVP because:
- **Enterprise contract required** — multi-month onboarding with a
  SATIM-supporting Algerian bank.
- Same lack of recurring / card-on-file tokenization as Chargily Pay — no
  functional advantage for MVP.
- Higher operational complexity (merchant account, bank relationship,
  per-bank sandbox credentials at `test2.satim.dz`).

SATIM direct remains an option **post-MVP** when monthly volume justifies
the enterprise contract overhead and the operator wants a branded payment
page.

### SlickPay / Mizaniya Pay as primary

Algerian alternative gateways with custom (quote-based) pricing. Listed as
**backup adapters** in the strategy pattern; not chosen as primary because
their pricing is opaque without a quote, whereas Chargily Pay's tiered
pricing is published.
