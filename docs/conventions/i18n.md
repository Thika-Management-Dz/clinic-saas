# i18n Conventions

- **Audience:** AI coding agents and human contributors
- **Blueprint references:** §8.1, §8.2
- **Source of truth for terminology:** `docs/domain/glossary.md`

## 1. Stack

| Concern | Choice |
|---|---|
| Library | **next-intl** (App-Router-native, ICU MessageFormat) |
| Locales | `ar-DZ` (RTL, Arabic, Western-Arabic numerals), `fr-DZ` (LTR, French) |
| Numerals | `numberingSystem: "latn"` for **both** locales (see §3) |
| Pluralization | ICU plurals — Arabic has 6 plural forms (zero, one, two, few, many, other) |
| Message files | `apps/web/messages/{ar-DZ,fr-DZ}.json` |

## 2. The single rule

> **Every user-visible string must go through `t()`.**

A string is "user-visible" if a clinic staff member or patient can read it on
screen: button labels, form labels, error messages, empty states, toast
notifications, table headers, chart legends, accessibility labels, email/SMS
templates, PDF invoice line descriptions.

If a string is not in `apps/web/messages/{locale}.json`, it does not exist.
**No hardcoded user-visible strings.** This is in the AGENTS.md Do-NOT list
and enforced by an ESLint rule (`eslint-plugin-no-hardcoded-strings` or
equivalent).

## 3. Numerals — Western Arabic for both locales

Both `ar-DZ` and `fr-DZ` use **Western Arabic numerals** (`0123456789`), not
Eastern Arabic numerals (`٠١٢٣٤٥٦٧٨٩`). This is the Algerian convention:
Algerian Arabic uses Western numerals in commerce, medicine, and
administration.

```ts
// next-intl config
{
  locale: 'ar-DZ',
  numberingSystem: 'latn',  // forces Western numerals even in ar-DZ
  // …
}
```

Numbers (prices, doses, tooth numbers, phone numbers, dates) render
identically in both locales. Only the surrounding text and the direction
flip.

## 4. Message-key namespacing

Keys are **namespaced by feature module**, mirroring the NestJS module tree
(ADR-002). This keeps the message file navigable and prevents key collisions
as the app grows.

```jsonc
// apps/web/messages/ar-DZ.json
{
  "common": {
    "save": "حفظ",
    "cancel": "إلغاء",
    "delete": "حذف",
    "confirm": "تأكيد",
    "search": "بحث"
  },
  "patients": {
    "list": {
      "title": "قائمة المرضى",
      "empty": "لا يوجد مرضى بعد"
    },
    "form": {
      "familyName": "اللقب",
      "givenName": "الاسم",
      "dob": "تاريخ الميلاد",
      "phone": "رقم الهاتف",
      "nin": "رقم التعريف الوطني"
    },
    "errors": {
      "phoneInvalid": "رقم الهاتف غير صالح",
      "ninDuplicate": "رقم التعريف الوطني مستعمل لمريض آخر"
    }
  },
  "appointments": { /* … */ },
  "encounters":  { /* … */ },
  "dental":      { /* … */ },
  "billing":     { /* … */ }
}
```

```jsonc
// apps/web/messages/fr-DZ.json — SAME key tree, French values
{
  "common": { "save": "Enregistrer", "cancel": "Annuler", /* … */ },
  "patients": {
    "list": { "title": "Liste des patients", "empty": "Aucun patient pour le moment" },
    /* … */
  }
  /* … */
}
```

### Key rules

- Mirror the **exact same key tree** across `ar-DZ.json` and `fr-DZ.json`.
  A key present in one and missing in the other is a CI error (a
  `scripts/check-i18n-keys.ts` script in CI asserts parity).
- Use `camelCase` for keys (e.g., `familyName`, not `family_name` or
  `family-name`).
- Namespaces match NestJS module names: `patients.*`, `appointments.*`,
  `encounters.*`, `dental.*`, `billing.*`, `inventory.*`, `imaging.*`,
  `audit.*`, `clinic.*`, `auth.*`, `common.*`.

## 5. ICU MessageFormat — plurals, genders, lists

Arabic has **6 plural forms** (zero, one, two, few, many, other). Use ICU
plurals — never build plural strings by hand.

```jsonc
// ar-DZ.json
{
  "appointments": {
    "list": {
      "count": "{count, plural, =0 {لا مواعيد} one {موعد واحد} two {موعدان} few {{count} مواعيد} many {{count} موعدًا} other {{count} موعد}}"
    }
  }
}

// fr-DZ.json
{
  "appointments": {
    "list": {
      "count": "{count, plural, =0 {Aucun rendez-vous} one {# rendez-vous} other {# rendez-vous}}"
    }
  }
}
```

```tsx
const t = useTranslations('appointments.list');
return <p>{t('count', { count })}</p>;
```

## 6. Dates, times, money

- **Dates**: use `next-intl`'s `format.dateTime` with locale-aware formatters.
  Algeria uses `DD/MM/YYYY` in French and the same numeric form in Arabic.
- **Times**: 24-hour clock (`14:30`), not 12-hour AM/PM.
- **Money**: DZD with `format.number({ style: 'currency', currency: 'DZD' })`.
  The currency column on `invoice` defaults to `'DZD'` and is a real column,
  not a hardcoded string (Blueprint §9.6).

## 7. Terminology — the glossary is the source of truth

`docs/domain/glossary.md` defines the canonical FR + AR + EN terms for every
domain concept (patient, encounter, prescription, odontogram, …). Message
values MUST use the glossary terms verbatim — no synonyms, no paraphrases.

Example: the glossary says `encounter` = `FR: consultation | AR: استشارة`.
So in `ar-DZ.json` we use `استشارة` for the encounter concept — never
`زيارة` (visit) or `لقاء` (meeting). Consistency across the UI is a clinical
safety property: a receptionist switching between locales must see the same
concept under the same name family.

## 8. Pseudo-localization in CI

A CI step runs the app under a pseudo-locale (`pseudo-LOL`) that wraps every
string in accents and pads it by ~30%, catching:

- Hardcoded strings (they don't get padded → visible in screenshot diff).
- Layout that breaks under 30% text expansion (a real risk for AR → FR
  translation growth).

## 9. Anti-patterns

| ❌ Don't | ✅ Do |
|---|---|
| `<button>Save</button>` | `<button>{t('common.save')}</button>` |
| `throw new Error('Phone invalid')` (user-visible) | `throw new ValidationError({ code: 'phoneInvalid', message: t('patients.errors.phoneInvalid') })` |
| `const title = 'Patient List'` | `const title = t('patients.list.title')` |
| Eastern Arabic numerals in ar-DZ | `numberingSystem: 'latn'` forces Western numerals |
| `patient` in ar-DZ, `client` in fr-DZ for the same concept | Both locales use the glossary term (`patient` / `مريض`) |
| Building plural strings by string concat | ICU `{count, plural, …}` |
