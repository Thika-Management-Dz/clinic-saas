2. Architecture Review
Verdict: architecturally sound. The hardest decisions are made correctly.
The architecture is the strongest part of this codebase. The operator has clearly thought through multi-tenancy,
modular monolith boundaries, offline-first sync, Algerian regulatory constraints, and the AI-agent development
workflow before writing a line of business logic. The 10 ADRs are not boilerplate — each one documents a real
decision with alternatives considered and rejected for stated reasons. ADR-001 (RLS pool model) in particular is
one of the better multi-tenancy ADRs I have read in an early-stage SaaS: it cites AWS SaaS Factory and Crunchy
Data, documents the Citus sharding escape hatch with concrete trigger thresholds (>200 tenants, >5TB, >500ms
p95), and acknowledges the operational trade-offs.
2.1 Monorepo structure
pnpm workspaces + Turborepo. Four apps (web, api, worker, patient-portal-stub) and seven packages (db, auth,
contracts, ui, i18n, eslint-config, tsconfig). Cross-package imports are restricted to declared public APIs via
no-restricted-imports pattern @clinic-saas/*/src/**. This is the correct enforcement mechanism — the operator
previously tried eslint-plugin-import/no-internal-modules and correctly removed it in Task 16 after finding it
blocked legitimate declared subpaths like @clinic-saas/db/schema. The lesson learned is documented in code
comments. This is what disciplined refactoring looks like.
2.2 Multi-tenancy: pool model with RLS (ADR-001)
The decision to use a single shared schema with tenant_id on every row, enforced by PostgreSQL Row-Level
Security at the database engine, is correct for the stated scale (2–50 clinics, ~120 encounters/day/clinic). The
alternative — database-per-tenant or schema-per-tenant — would be operational suicide for a solo operator using
AI agents as the workforce. The RLS policy pattern is correct:
USING (tenant_id = NULLIF(current_setting('app.current_tenant', true), '')::uuid)
WITH CHECK (tenant_id = NULLIF(current_setting('app.current_tenant', true), '')::uuid)
The NULLIF + missing_ok pattern is the right defensive posture: if the application ever forgets to SET LOCAL
app.current_tenant before querying, the policy returns NULL, and tenant_id = NULL is false for all rows. The app
sees nothing rather than everything. This is the correct fail-closed behavior. The CI test in rls.test.ts explicitly
verifies this case.
2.3 Modular monolith (ADR-002)
NestJS modular monolith over microservices is the right call at this scale. One deployment, one DB connection
pool, one CI pipeline. Cross-module write-side effects via EventEmitter2 with JSON-serializable payloads (no class
instances) is a thoughtful future-proofing decision: it preserves the option to extract to a message bus later without
contract changes. The Do-NOT list item #10 ("Do NOT import another NestJS module's internal services — only
index.ts public APIs") is enforced at the lint layer, not just documented.
2.4 Dependency direction
Clean. apps/* depend on packages/*. packages/* do not depend on apps/*. Within packages, the dependency graph
is: db ← (auth, contracts); ui ← (none); i18n ← (none); eslint-config ← (none); tsconfig ← (none). No cycles.
eslint-plugin-import-x no-cycle is enabled and would catch any future cycle. The eslint config itself is .js (not .mjs)
which is a minor inconsistency with the apps' .mjs configs but functionally equivalent.
2.5 Architectural concerns

Three real concerns, none of which invalidate the architecture but all of which need attention before Phase 5:
(a) app_user.tenantId is nullable for super_admin. The schema comment acknowledges this is a security trap:
"the auth module (Phase 5) needs special handling" using a SECURITY DEFINER function or separate connection.
Until that lands, the schema is in an unstable state — any code that touches app_user today will hit the RLS NULL
trap. The login flow (find user by email without a tenant context) is not yet solved.
(b) Audit log hash chain is global, not per-tenant. The compute_audit_hash_curr() trigger function reads
MAX(id) across all tenants. This means tenant A's audit log integrity depends on tenant B's audit log integrity. A
more correct design would be per-tenant hash chains (filter by tenant_id when reading prev_hash). The current
design is tamper-evident globally but not per-tenant.
(c) No optimistic concurrency columns. The blueprint mentions updated_at for Last-Write-Wins conflict
resolution in offline sync (Phase 8). But LWW is the weakest conflict resolution strategy. For clinical data,
optimistic locking via a version column or client-side conflict UI would be safer. The schema does not currently
include version columns — adding them later requires a migration that touches every tenant-scoped table.

3. Security Review — P0 Findings (Blockers)
Six findings must be resolved before this codebase is allowed to touch real patient data. Each is documented with
file:line, code excerpt, root cause, and remediation. None of them are subtle — they are all visible from a careful

3.1 P0-1: No CI/CD workflows exist
Location: .github/workflows/ directory does not exist.
Severity: P0 (process blocker).
Root cause: ADR-010 explicitly defers CI to "Phase 7" and relies on the author's local pnpm lint && pnpm
typecheck && pnpm test as the merge gate. README states these "must pass" before pushing a PR. Nothing
enforces this.
Risk: a single missed lint/typecheck/test locally ships broken code to main. AGENTS.md's Do-NOT list (12 items)
is not enforced by any machine. The compliance-gate RLS tests in packages/db/src/__tests__/rls.test.ts require a
real Postgres connection — without CI, no one is running them on every PR.
Remediation: add a .github/workflows/ci.yml that runs on PR open and on push to main, with jobs for lint,
typecheck, unit test, and (with a Postgres service container) integration test. Pin actions to SHA, not tag. Add a
branch protection rule requiring the CI check to pass before merge. This is a 1-day task and unblocks every other
process improvement.
3.2 P0-2: Dev DB credentials hardcoded in committed SQL and compose
Location: packages/db/sql/001_roles.sql:36,42; docker-compose.yml:44; .env.example:68,75

The comments are honest: "These are NOT secrets — they are committed because the dev environment is
non-sensitive." This is defensible for local dev. The problem is operational: there is no documented upgrade path.
The Docker entrypoint only runs 001_roles.sql on first init (empty data dir). On an existing volume, the passwords

contains the literal string "dev_password"; (c) document the staging/prod rotation procedure in docs/runbooks/.
This is a 2-hour fix.
3.3 P0-3: SQL injection pattern in withTenant test helper
Location: packages/db/src/__tests__/helpers.ts:87

sql: postgres.Sql,
tenantId: string,
fn: (tx: postgres.TransactionSql) => Promise<T>,

): Promise<T> {
const result = await sql.begin(async (tx) => {
await tx.unsafe(`SET LOCAL app.current_tenant = '${tenantId}'`);
return await fn(tx);
});
return result as T;

The tenantId is string-interpolated into a SET LOCAL statement via tx.unsafe(). Today this is safe because tests use
hardcoded UUIDs. But this helper is the obvious template for the Phase 5 TenantInterceptor. If that interceptor
copies this pattern (and it likely will, because the helper is the only reference implementation), then any
user-controlled value flowing into the tenant context becomes a SQL injection vector. SET LOCAL does not accept
parameters in postgres.js's tagged-template syntax, which is why the author used unsafe() — but the correct pattern
is to validate tenantId as a UUID before interpolation, or to use a SECURITY DEFINER function that takes the
tenant as a parameter.
Remediation: add a UUID regex check at the top of withTenant (throw if tenantId does not match
/^[0-9a-f-]{36}$/i). Document this requirement in the Phase 5 TenantInterceptor spec. Better: use a parameterized
PL/pgSQL function set_tenant(p_tenant uuid) that does the SET LOCAL internally.
3.4 P0-4: Audit log hash chain race condition
Location: packages/db/sql/002_audit_log_immutable.sql:95-98
CREATE OR REPLACE FUNCTION compute_audit_hash_curr()
RETURNS TRIGGER LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$
DECLARE prev_hash TEXT;
BEGIN
SELECT hash_curr INTO prev_hash FROM audit_log ORDER BY id DESC LIMIT 1;
NEW.hash_prev := prev_hash;
-- ... compute hash_curr ...
RETURN NEW;
END;
$$;
Two concurrent INSERTs in separate transactions can both read the same MAX(id) row's hash_curr. Both new
rows will then have hash_prev pointing to the same predecessor, and both will compute their hash_curr from the
same input. When the transactions commit, the chain has a fork: the row with the higher id has hash_prev equal to
the predecessor's hash_curr, but the predecessor's hash_curr is no longer the immediately-preceding row. The
"chain links" test in audit_log.test.ts:141 only verifies row N's hash_prev == row N-1's hash_curr — it does not test
concurrent inserts.
At 120 encounters/clinic/day × multiple audit events per encounter × 50 clinics, concurrent audit_log INSERTs are
guaranteed in production. The race will fire. The chain will break. The breach-response runbook does not cover this
case.
Remediation: (a) add LOCK TABLE audit_log IN SHARE ROW EXCLUSIVE MODE inside the trigger function
(serializes INSERTs, costs throughput); (b) use an advisory lock
pg_advisory_xact_lock(hashtext('audit_log_chain')) (cheaper); (c) redesign as per-tenant hash chains with tenant_id
in the prev_hash lookup (eliminates cross-tenant contention); (d) use a sequence + hash_prev = explicit lookup by
id = currval() - 1. Option (c) is recommended — it also fixes P0-5.
3.5 P0-5: SECURITY DEFINER function reads across all tenants

P0 BLOCKER P1 HIGH P2 MEDIUM P3 LOW OK STRENGTH
Location: packages/db/sql/002_audit_log_immutable.sql:82-132 (function), :149-153 (trigger)
The compute_audit_hash_curr() function is declared SECURITY DEFINER, meaning it runs with the privileges of
the function owner (postgres superuser on local dev, neondb_owner on Neon staging). It executes a SELECT
against audit_log ORDER BY id DESC LIMIT 1 — this read bypasses RLS because of SECURITY DEFINER.
The comment acknowledges this: "the function only reads hash_curr (a hash, not PII) — no patient data is
exposed."
That is correct as far as it goes. The deeper problem is architectural: tenant A's audit log chain is now coupled to
tenant B's. If tenant B's data is corrupted (failed INSERT, manual fix by ops_superuser, partial restore from
backup), tenant A's chain breaks too. A clinic being audited by ANPDP cannot independently verify the integrity of
its own audit log — it must trust the global chain. This is a compliance smell, even if it is not strictly a security
vulnerability.
Remediation: redesign as per-tenant hash chains. The trigger function becomes:
SELECT hash_curr INTO prev_hash
FROM audit_log
WHERE tenant_id = NEW.tenant_id
ORDER BY id DESC LIMIT 1;
This is RLS-safe within SECURITY DEFINER (the function owner has BYPASSRLS), per-tenant verifiable, and
eliminates cross-tenant contention. It also fixes the race condition for any single tenant (concurrent INSERTs from
the same tenant are still racy, but those are rare — one tenant's audit events come from one clinic's users, mostly
serially).
3.6 P0-6: app_user nullable tenant_id without super_admin login flow
Location: packages/db/src/schema/app_user.ts:32, comment :14-17
app_user.tenantId is nullable (for super_admin cross-tenant accounts). The RLS policy is tenant_id =
NULLIF(current_setting(...)). When current_tenant is set, only users with that tenant_id are visible. When
current_tenant is NOT set, tenant_id = NULL is false, so NO users are visible — including super_admin. This is
the correct fail-closed posture, but it means there is currently no way to log in as super_admin. The schema
comment acknowledges: "Phase 5's login flow needs special handling — a SECURITY DEFINER function or a
separate connection that sets current_tenant to a special system value."
Until Phase 5 lands, the schema is in an unstable state. Any code that touches app_user (login, user CRUD,
user_role assignment) will hit the RLS NULL trap. The seed.ts script sidesteps this by connecting as ops_superuser
(BYPASSRLS), but the application cannot. This is a known, documented trap — the issue is that the trap exists at
all. A more defensive design would have a separate app_user_login table (no RLS, no tenant_id) holding just email
+ password_hash, with app_user being tenant-scoped profile data. Login queries the login table; session creation
queries the tenant-scoped table after the tenant is resolved.

5. Data Model & RLS Review
Verdict: correct where it matters, sloppy where it does not.
The RLS posture is genuinely impressive for a Phase 4 codebase. The compliance-gate tests in

10. Final Verdict & Recommendations
This is a frustrating review to write, because the project is genuinely well-architected and the operator's discipline is
unusual. The ADRs are real, the AGENTS.md is exemplary, the RLS posture is correct, the compliance-gate tests
are the kind most teams never write. If every early-stage SaaS had this level of architectural forethought, the
industry would be in a better place.
But the codebase is not production-ready, and the operator's own ADR-010 acknowledges the biggest reason: no CI,
no machine-enforced merge gate, relying on author self-verification. For a healthcare SaaS processing patient data
under Algerian Law 18-07, that is unacceptable. The six P0 blockers are not subtle — they are visible from a
careful read of the code — but they will ship to main if nothing enforces the rules. ADR-010's decision to replace
an always-on AI PR review bot with a manual AI agent review session is defensible on cost and egress grounds, but
it removes a machine gate without replacing it with another machine gate. The manual session is author-blind
review; it is not author-blind enforcement. The difference matters.
Three recommendations, in priority order:
1. Do not deploy to production until the 30-day blockers are resolved. The 30-day roadmap is 5 days of work
for a single engineer. None of it is architecturally hard — it is process hygiene (CI, branch protection, secrets
rotation, hash chain fix, SECURITY.md). Until this lands, the codebase is a well-designed skeleton that cannot be
trusted with patient data.
2. File the DPIA and appoint a DPO before any real patient data touches the system. The DPIA stub is honest
about the Law 25-11 sanctions: Art. 65 fine 200K-500K DZD; Art. 58 sensitive-data violation: 6 months to 1 year
imprisonment. The ANPDP filing process is multi-month. Start it now, in parallel with Phase 5 implementation.
The DPIA itself is more thorough than most filed DPIAs I have seen — completing it is a documentation task, not a
research task.
3. Address the bus factor before go-live. CODEOWNERS = @AguHo only. Every line of code, every ADR,
every operational runbook, every merge decision is one person. For a healthcare SaaS, this is the single largest
operational risk — larger than any of the P0 blockers, because it cannot be fixed by a code change. Hire or contract
a second engineer with commit access before go-live. If that is not possible, document the runbooks exhaustively
(already in progress), record Loom videos of common operational tasks, and consider open-sourcing to attract
contributors. The AGENTS.md is excellent contributor onboarding — use it.
What to keep doing: the AGENTS.md + ADR discipline is the project's strongest asset. The phase-by-phase
roadmap with PR-per-task granularity, the worklog as audit trail, the explicit Do-NOT list with rationales, the
compliance-gate tests that throw rather than skip — these are the practices that separate this project from the 95%
of early-stage SaaS codebases that collapse under their own weight. Keep doing all of it. The codebase will be
production-ready when the process catches up to the architecture.
